# Dating App — Бизнес-логика сервисов

## 1. Auth Service (:8001)

**Назначение:** Аутентификация и авторизация пользователей.

### Функции

- **Регистрация** — создание аккаунта по email/телефону, хэширование пароля, отправка кода подтверждения
- **Логин** — проверка credentials, выдача JWT access + refresh токенов
- **Обновление токенов** — ротация refresh-токена, инвалидация старого
- **Верификация email/телефона** — подтверждение через код (SMS/email)
- **Сброс пароля** — генерация одноразовой ссылки, смена пароля
- **Logout** — инвалидация сессии в Redis

### Хранилища

- **PostgreSQL (auth_db):** таблица `accounts` — email, phone, password_hash, статусы
- **Redis:** активные сессии и refresh-токены (с TTL)

### Взаимодействие

- API Gateway валидирует JWT через Auth Service при каждом запросе
- При регистрации отправляет событие в Kafka → User Service создаёт запись в `users`

---

## 2. User Service (:8002)

**Назначение:** Управление пользователями, подписками, блокировками и жалобами.

### Функции

- **Управление пользователем** — получение/обновление настроек, статус аккаунта (active/banned/deleted)
- **Реферальная программа** — генерация уникального реферального кода, начисление бонусов при регистрации по коду (premium-дни, super likes)
- **Подписки (Premium)** — активация/деактивация premium, проверка `premium_until`
- **Блокировки** — добавление пользователя в блок-лист, проверка блокировок при свайпах и в ленте
- **Бан** — блокировка аккаунта с указанием причины

### Хранилища

- **PostgreSQL (user_db):** таблицы `users`, `referral_bonuses`, `user_blocks`, 

### Взаимодействие

- Feed Service запрашивает список заблокированных пользователей для исключения из ленты
- Matching Service проверяет блокировки перед обработкой свайпа
- Получает событие регистрации из Kafka → создаёт запись в `users`

---

## 3. Profile Service (:8003)

**Назначение:** Управление анкетами, фотографиями и поисковыми предпочтениями.

### Функции

- **CRUD анкеты** — имя, дата рождения, пол, био, город, рост, образование, профессия
- **Управление фото** — загрузка (→ Image Processor → MinIO), удаление, смена порядка, установка главного фото
- **Интересы** — добавление/удаление интересов из справочника, привязка к профилю
- **Поисковые предпочтения** — возраст (min/max), пол, максимальная дистанция, фильтры (только верифицированные, только с био)
- **Верификация профиля** — загрузка верификационного фото, сравнение с основными фото
- **Геолокация** — обновление координат (PostGIS), определение города/страны
- **Completion score** — расчёт полноты анкеты (заполнены ли фото, био, интересы и т.д.)

### Хранилища

- **PostgreSQL (profile_db):** таблицы `profiles`, `interests`, `profile_interests`, `photos`, `search_preferences`
- **MinIO (S3):** хранение оригиналов и обработанных фото

### Взаимодействие

- Feed Service запрашивает профили для формирования ленты
- AI Service читает данные профиля для анализа и генерации embeddings
- Image Processor обрабатывает загруженные фото (ресайз, WebP, NSFW-детекция)
- При обновлении профиля публикует событие `profile.updated` в Kafka → Rating Service пересчитывает profile_score

---

## 4. Matching Service (:8004)

**Назначение:** Обработка свайпов, определение мэтчей, управление лимитами.

### Функции

- **Свайп (like/dislike/super_like)** — сохранение действия, проверка на взаимный лайк
- **Определение мэтча** — если оба пользователя поставили like/super_like → создание мэтча, создание чат-комнаты
- **Rewind** — отмена последнего свайпа (premium-функция), проверка лимита rewind'ов
- **«Кто меня лайкнул»** — список pending likes (premium: видны все, free: только количество)
- **Daily limits** — отслеживание и проверка лимитов super likes, rewinds, boosts за день
- **Unmatch** — деактивация мэтча, закрытие чат-комнаты

### Хранилища

- **PostgreSQL (matching_db):** таблицы `swipes`, `matches`, `pending_likes`, `daily_limits`

### Взаимодействие

- Проверяет блокировки через User Service перед обработкой свайпа
- Публикует события в Kafka: `swipe.created`, `match.created`, `match.unmatched`
- Rating Service получает эти события для обновления поведенческих метрик
- Notification Service получает `match.created` → push-уведомление обоим пользователям
- Chat Service создаёт комнату при новом мэтче

---

## 5. Rating Service (:8005)

**Назначение:** Расчёт и хранение рейтингов пользователей для ранжирования в ленте.

### Функции

- **Первичный рейтинг (Profile Score)** — оценка полноты и качества анкеты: наличие фото, длина био, количество интересов → `primary_rating` (0–100)
- **Поведенческий рейтинг (Behavioral Score)** — на основе входящих событий из Kafka: like_ratio, match_rate, chat_initiation_rate, response_rate → `behavioral_rating` (0–100)
- **Комбинированный рейтинг (Combined Score)** — взвешенная сумма primary + behavioral + бонусы (реферальный, premium) → `combined_rating` (0–100)
- **Периодический пересчёт** — Celery Beat запускает полный пересчёт рейтингов по расписанию
- **История изменений** — запись каждого изменения рейтинга с причиной и старым/новым значением
- **Дневная статистика** — агрегация impressions, likes, matches за день для аналитики

### Хранилища

- **PostgreSQL (rating_db):** таблицы `user_ratings`, `rating_history`, `rating_stats_daily`
- **Redis:** кэш текущих рейтингов для быстрого доступа из Feed Service

### Взаимодействие

- Потребляет события из Kafka: свайпы, мэтчи, обновления профиля → пересчитывает соответствующие метрики
- Feed Service запрашивает рейтинги для сортировки ленты
- Celery + Beat запускает периодический пересчёт

---

## 6. Feed Service (:8006)

**Назначение:** Формирование персонализированной ленты профилей для свайпов.

### Функции

- **Генерация ленты** — выборка кандидатов на основе поисковых предпочтений (возраст, пол, дистанция)
- **Фильтрация** — исключение: уже просмотренных (свайпнутых), заблокированных пользователей, собственного профиля
- **Сортировка по рейтингу** — кандидаты ранжируются по `combined_rating` из Rating Service
- **Кэширование** — подготовленная лента кэшируется в Redis, обновляется при исчерпании или по событию

### Хранилища

- **Redis:** кэш подготовленной ленты для каждого пользователя

### Взаимодействие

- Читает профили и search_preferences из Profile Service (PostgreSQL)
- Запрашивает рейтинги из Rating Service
- Запрашивает список блокировок из User Service
- Получает события из Kafka (rating changes) для инвалидации кэша

---

## 7. Chat Service (:8007)

**Назначение:** Обмен сообщениями между пользователями с мэтчем.

### Функции

- **WebSocket-соединение** — установка и поддержание real-time соединения для каждого пользователя
- **Отправка сообщений** — текст, изображения, GIF, голосовые (проверка наличия активного мэтча)
- **Read receipts** — отметка о прочтении, уведомление отправителя
- **Typing indicators** — индикатор набора текста через WebSocket (состояние в Redis)
- **Онлайн-статус** — отслеживание online/offline/last_seen через Redis
- **Удаление сообщений** — мягкое удаление (is_deleted), скрытие из выдачи
- **История сообщений** — пагинация по cursor (по дате), загрузка старых сообщений

### Хранилища

- **PostgreSQL (chat_db):** таблицы `chat_rooms`, `messages`, `typing_events`, `read_receipts`
- **Redis:** онлайн-статусы, typing-индикаторы, кэш последних сообщений

### Взаимодействие

- Проверяет наличие активного мэтча через Matching Service перед отправкой сообщения
- Публикует события в Kafka: `message.sent`, `message.read` → Notification Service отправляет push
- При создании мэтча (событие из Kafka) автоматически создаёт чат-комнату

---

## 8. Notification Service (:8008)

**Назначение:** Отправка уведомлений пользователям по всем каналам.

### Функции

- **Push-уведомления** — отправка через FCM (Android) / APNs (iOS) при новом мэтче, сообщении, лайке
- **Email-уведомления** — welcome-письмо, подтверждение email, еженедельный дайджест активности
- **In-App уведомления** — бейджи, счётчики непрочитанных, список уведомлений в приложении
- **Управление предпочтениями** — пользователь выбирает какие уведомления получать и по каким каналам
- **Дедупликация** — предотвращение повторной отправки одного и того же уведомления

### Хранилища

- **Redis:** кэш токенов устройств, настроек уведомлений, очередь отправки

### Взаимодействие

- Потребляет события из Kafka: мэтчи, сообщения, лайки, системные события
- Отправляет через внешние сервисы: FCM, APNs, SMTP, SMS-провайдер

---

## 9. AI Service (:8010)

**Назначение:** AI-анализ профилей, рекомендации по улучшению, семантический поиск.

### Функции

- **Анализ профиля** — оценка качества фото (освещение, улыбка, видимость лица), качества био (длина, тон, уникальность), интересов → общий score (0–100)
- **Рекомендации по улучшению** — персональные советы: «загрузите фото на улице», «добавьте больше интересов», «перепишите био» с приоритетами (high/medium/low)
- **Conversation starters** — генерация фраз для начала разговора на основе общих интересов и профилей обоих пользователей (funny/thoughtful/flirty стили)
- **Семантический поиск** — генерация embeddings профилей (OpenAI text-embedding-3-small, 1536-dim), сохранение в Qdrant, поиск похожих профилей по косинусному расстоянию
- **Аудит AI-запросов** — логирование каждого вызова LLM: модель, токены, стоимость, латентность
- **Feedback** — сбор обратной связи от пользователей на рекомендации (1–5 звёзд, полезность)

### Хранилища

- **PostgreSQL (ai_db):** таблицы `profile_analyses`, `profile_recommendations`, `conversation_starters`, `ai_requests`, `ai_feedback`
- **Qdrant (Vector DB):** коллекция `profiles` — embeddings профилей с payload (user_id, gender, age, city) и фильтрацией

### Взаимодействие

- Читает данные профилей из Profile Service для анализа
- Celery Workers выполняют асинхронный анализ (тяжёлые LLM-вызовы)
- Feed Service может использовать семантический поиск для улучшения ленты
- При обновлении профиля (событие из Kafka) — пересоздаёт embedding в Qdrant
